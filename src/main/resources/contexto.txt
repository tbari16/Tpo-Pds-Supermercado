# Context Prompt for Backend Development — E-Market Supermarket (University Project)

## Project Overview

You are helping me build the backend of a university project called **E-Market**, an online supermarket e-commerce application. This is an academic project (TPO) for the subject "Software Development Process" at UADE, Argentina.

The backend must be built with:
- **Java + Spring Boot**
- **Spring Data JPA + Hibernate**
- **MySQL** as the relational database
- **JWT** for authentication and authorization
- **Layered architecture**: Controllers, Services, Repositories, Entities/Models, DTOs, Exceptions

The project has a React + Vite frontend that will consume the REST API, but you only need to help with the backend.

---

## Architecture and Folder Structure

The project must follow this package structure:

```
com.emarket
├── controllers        → REST controllers, no business logic here, only delegation to services
├── services           → Business logic, design patterns live here and in domain
├── repositories       → Spring Data JPA interfaces
├── domain             → Pure domain classes (State pattern objects, Strategy objects, Observer objects)
├── entities           → JPA entities mapped to database tables
├── dto                → Data Transfer Objects for request/response
├── exceptions         → Custom domain exceptions
├── config             → Spring Security config, JWT config, Bean declarations
└── factory            → Factory classes (MetodoPagoFactory)
```

**Critical rule**: Controllers must never contain business logic. They receive HTTP requests, validate input via DTOs, and delegate everything to services. Services contain all business logic and orchestrate domain objects.

## Design Patterns Implementation

### 1. Strategy Pattern — Payment Methods

**Where**: `domain/payment/` package and `services/PagoService.java`

**Structure**:
```java
// Strategy interface
public interface MetodoPago {
    ResultadoPago procesarPago(BigDecimal monto, String referenciaPedido);
    MetodoPagoEnum getTipo();
}

// Concrete strategies — each is a @Component
@Component public class PagoTarjetaCredito implements MetodoPago { ... }
@Component public class PagoMercadoPago implements MetodoPago { ... }
@Component public class PagoTransferencia implements MetodoPago { ... }

// Factory — receives all strategies injected by Spring automatically
@Component
public class MetodoPagoFactory {
    private final Map<MetodoPagoEnum, MetodoPago> estrategias;
    public MetodoPagoFactory(List<MetodoPago> estrategias) {
        this.estrategias = estrategias.stream()
            .collect(Collectors.toMap(MetodoPago::getTipo, Function.identity()));
    }
    public MetodoPago crearMetodoPago(MetodoPagoEnum tipo) { ... }
}

// Context
@Service
public class PagoService {
    private final MetodoPagoFactory factory;
    public ResultadoPago ejecutarPago(MetodoPagoEnum tipo, BigDecimal monto, String ref) {
        return factory.crearMetodoPago(tipo).procesarPago(monto, ref);
    }
}
```

**ResultadoPago** is a plain Java class (not an entity):
- exitoso: boolean
- referenciaPago: String
- mensaje: String
- fechaProcesamiento: LocalDateTime

### 2. State Pattern — Order Lifecycle

**Where**: `domain/state/` package

**Structure**:
```java
// State interface
public interface EstadoPedido {
    void pagar(Pedido pedido);
    void enviar(Pedido pedido);
    void entregar(Pedido pedido);
    void cancelar(Pedido pedido);
    EstadoPedidoEnum getNombre();
}

// Concrete states — plain Java classes, NOT Spring beans, NOT JPA entities
public class EstadoPendiente implements EstadoPedido {
    @Override
    public void pagar(Pedido pedido) {
        pedido.cambiarEstado(new EstadoPagado()); // triggers state change + notification
    }
    @Override
    public void cancelar(Pedido pedido) {
        pedido.cambiarEstado(new EstadoCancelado());
    }
    @Override
    public void enviar(Pedido pedido) {
        throw new InvalidStateTransitionException("No se puede enviar un pedido pendiente");
    }
    @Override
    public void entregar(Pedido pedido) {
        throw new InvalidStateTransitionException("No se puede entregar un pedido pendiente");
    }
}
// EstadoPagado, EstadoEnviado, EstadoEntregado, EstadoCancelado follow same pattern
```

**Important**: `Pedido.estadoActual` is `@Transient`. When a `Pedido` is loaded from the database, the service layer reconstructs the state object from `estadoEnum` before operating:
```java
pedido.setEstadoActual(EstadoFactory.from(pedido.getEstadoEnum()));
```

Valid transitions:
- PENDIENTE → PAGADO or CANCELADO
- PAGADO → ENVIADO or CANCELADO
- ENVIADO → ENTREGADO
- ENTREGADO → no transitions allowed
- CANCELADO → no transitions allowed

### 3. Observer Pattern — Notifications

**Where**: `domain/observer/` package and `services/NotificacionService.java`

**Structure**:
```java
// Observable interface (subject)
public interface Observable {
    void agregarObservador(ObservadorPedido obs);
    void eliminarObservador(ObservadorPedido obs);
    void notificar();
}

// Observer interface
public interface ObservadorPedido {
    void actualizar(Pedido pedido, EstadoPedidoEnum nuevoEstado);
}

// Concrete observers — each is a @Component
@Component public class EmailObservador implements ObservadorPedido { ... }
@Component public class SMSObservador implements ObservadorPedido { ... }
@Component public class PushObservador implements ObservadorPedido { ... }
```

`Pedido` implements `Observable`. The `observadores` list is `@Transient`.
`NotificacionService` is responsible for injecting the observers into a `Pedido` before state changes occur, and for persisting `Notificacion` entities after notification.

### 4. Composite Pattern — Category Catalog

**Where**: `entities/CategoriaComponente.java`, `entities/CategoriaHoja.java`, `entities/CategoriaCompuesta.java`

Use JPA `@Inheritance(strategy = InheritanceType.JOINED)` for the hierarchy.
`CategoriaCompuesta` has a `@OneToMany` self-referential relationship to `CategoriaComponente`.
`Producto` has `@ManyToOne` to `CategoriaComponente`.

The key behavior is that `listarProductos()` on a `CategoriaCompuesta` recursively collects products from all its children, treating leaves and composites uniformly.

### 5. Facade Pattern — Checkout Process

**Where**: `services/CheckoutFacade.java`

```java
@Service
public class CheckoutFacade {
    // Injects: PagoService, NotificacionService, CarritoService, PedidoService
    
    public Pedido confirmarCompra(Cliente cliente, MetodoPagoEnum tipoPago, String direccionEnvio) {
        // 1. Validate cart is not empty
        // 2. Validate stock for each item
        // 3. Execute payment via PagoService
        // 4. Create Pedido with EstadoPendiente
        // 5. Register observers via NotificacionService
        // 6. Trigger pagar() on pedido → changes state to PAGADO → notifies observers
        // 7. Reduce stock for each product
        // 8. Clear cart
        // 9. Return pedido
    }
}
```

## Important Design Decisions to Respect

1. **State objects are never JPA entities**. They are plain Java objects reconstructed from the `estadoEnum` column every time a `Pedido` is loaded and needs to perform a state operation.

2. **Observer list in Pedido is @Transient**. Observers are Spring beans injected by `NotificacionService`, not persisted.

3. **Strategy selection happens at runtime**. `MetodoPagoFactory` uses a Map built from Spring-injected beans, so adding a new payment method only requires creating a new `@Component` that implements `MetodoPago`.

4. **Composite recursion is in domain methods**. `CategoriaCompuesta.listarProductos()` calls `listarProductos()` on each child, letting the tree resolve itself. Services should not reproduce this recursion.

5. **Controllers only delegate**. No `if` statements for business logic in controllers. Input validation via `@Valid` and DTOs only.

6. **Soft delete for products**. `bajaProducto()` sets `activo = false`, never deletes from database. `CatalogoService` always filters by `activo = true`.

7. **Repositories are the only DIP bridge**. Services depend on Repository interfaces (Spring Data JPA), never directly on entity constructors for persistence operations. This respects Dependency Inversion even though Repositories are not shown in the class diagram.

8. **Price capture on cart and order creation**. `CarritoItem.precioUnitario` and `PedidoItem.precioUnitario` capture the product price at the moment of adding, not a reference to the current price. This prevents price changes from affecting existing carts or orders.
